Creative direction map

Choose from the whole bench.

The landing page should preview the operating model: use the live repository list first, pull in specialist workflows only when they sharpen the artifact, and keep external services optional until a concrete asset is needed.

Design files Canonical page first.

index.html is the canonical public page; CNAME binds hermegeddon.dev; README.md records local preview with Python's HTTP server.

Skills Direction before decoration.

Creative direction and design review are the useful lane here; media, deck, and ad skills wait until the page needs those surfaces.

Templates Optional export path.

Screenshot, accessibility, PDF, or hosting templates are useful for review and publishing, not for inventing new page content.

Connectors GitHub API as source.

The real-data pass uses public GitHub metadata only: repo names, descriptions, languages, licenses, fork/source status, Pages, and update dates.

Design posture

Useful, then public.

The page should make one promise: Hermegeddon publishes work with source-first context, visible boundaries, and enough provenance to be inspected later.
